# Bank Reconciliation SaaS - מערכת התאמות בנקאיות

## Overview
This project is a B2B SaaS application for automated bank reconciliation. It is a multi-tenant system designed to streamline the process of matching bank transfers to payment requests, managing unidentified transfers, and providing a robust platform for customer and bank account management. The system aims to automate payment request lifecycles, facilitate communication with customers regarding payments, and offer advanced reconciliation features. Its core purpose is to simplify financial operations for businesses by automating bank reconciliation and payment processing.

## System Architecture
The application features a React, TypeScript, shadcn/ui, and Tailwind CSS frontend with dynamic RTL/LTR support. The backend is built with Express.js following a layered architecture (Models, Services, Routes). MongoDB Atlas is used for data persistence, managed via Mongoose. The system supports internationalization (i18n) with Hebrew, English, Arabic, Russian, and Turkish.

Key architectural decisions include:
- **UI/UX**: Dynamic RTL/LTR layout based on selected language (HE/AR = RTL, EN = LTR) using a Zinc/Slate color theme. Sidebar navigation is placed on the right for RTL languages.
- **Multi-tenancy (LIVE)**: Database-per-tenant architecture. Each tenant has a dedicated MongoDB database on Atlas (`tenant_<slug>`). Central DB holds tenants, channels, users. All API routes use `req.tenantDbConnection` with factory models. Migration script: `server/scripts/run-production-migration.ts` — auto-discovers orphaned data, creates tenant records, migrates to isolated DBs. **Message Quotas**: `monthlyMessageQuota` (default 999999), `messagesUsedThisMonth` (outbound, default 0), `inboundMessagesThisMonth` (inbound, default 0). Flexi-quota: blocks only when `quota > 0 AND used >= quota`. **GDPR Data Purge**: `tenantService.purgeTenantData(tenantId)` drops all collections in tenant's isolated DB and sets `active: false` in central DB.
- **Authentication**: Tenant-scoped login via URL slug (e.g., `/login/herz`). LoginPage resolves slug via `GET /api/public/tenant/:slug`, displays company name, and passes `tenantId` in login/verify requests. Backend `findUserByIdentifier` scopes by `tenantId` when provided. Superadmin uses `/admin` route (no tenant context). `/login` without slug shows error. OTP via SMS or Email with tenant credentials (global fallback). Token-based sessions.
- **Bank Account Management**: A FIFO engine atomically assigns available bank accounts per tenant. Accounts are released upon payment completion, manual release, or expiry.
- **Payment Request Lifecycle**: Payment requests support cumulative payments and auto-complete when the amount received meets the requested amount. A background job handles overdue request expiration and bank account release.
- **Communication**: Payment links are shortened using Cuttly.
- **Reconciliation**: Webhooks receive transactions and attempt to auto-match them to active payment requests. Unmatched transactions create disputes that can be manually resolved via a dedicated "Unidentified Transfers" dashboard.
- **Customer Management**: A `Customer` entity linked to payment requests with a unified customer form supporting standard and dynamic custom fields. **Per-Tenant Custom Fields**: `customerFields: CustomFieldDefinition[]` on Tenant model — configurable via dedicated "Customer Fields Editor" dialog (accessible from tenant actions in Tenants page). Each field has: key, label, fieldType (text/number/date/select/combobox/textarea/boolean), uiWidth (3/4/6/9/12 grid columns), options (for select/combobox), isFilterable, forceNewRow, order. Drag-to-reorder support. API: `GET/PUT /api/tenants/:id/customer-fields`.
- **Email System**: Multi-tenant SendGrid integration with hierarchical credential management.
- **Audit Trail**: Significant actions like manual matching, transfer assignment, and bank account releases are logged with `actorId`/`actorName`.
- **Omnichannel Communication**: Unified data models for multi-channel communication (WhatsApp, SMS, Email) including `Customer`, `Conversation`, and `Message` models. Supports full media for WhatsApp.
- **Teams/Departments**: Team model (tenantId, name, description, color) with CRUD API. Users support multi-team assignment via `teamIds: string[]` array. Tags can be scoped to teams (optional `teamId` field). Inbox resolve dialog filters tags by agent's team memberships (includes global tags). Settings page provides Teams and Tags management UI.
- **Conversation Ownership & Routing**: Implements a CommBox-style conversation assignment system with `assignedTo`, `groupId`, and routing rules (Sticky Agent, Load Balancer - Round Robin, Pool). Includes agent presence (`isOnline`, `lastSeenAt`) and real-time updates via Socket.io. Routing service supports both legacy `groupId` and new `teamIds` array for agent matching.
- **Workflow Features**: Includes resolve dialog with tag input, summary, snooze system with time presets + custom date/time calendar picker and background job, and Quick Replies. Snooze supports target agent assignment: when snoozing, user can choose which agent the conversation will be assigned to upon wake (default: self). Snoozed conversations appear exclusively in the "Snoozed" tab (not in "Mine"). Fields: `snoozedBy`, `snoozeWakeAgentId`, `snoozeWakeAgentName` on Conversation model. Wake service validates target agent is active before reassignment.

## Multi-Tenant Database Architecture
- **Central DB** (`cpaas-platform`): Hosts Users, Tenants, and global mappings. Connected via `mongoose.connect()` in `server/db.ts`. Exported constant `CENTRAL_DB_NAME`.
- **Tenant DB URI**: `tenantDbUri` field on Tenant model (`select: false` for security) stores per-tenant database connection string for future database-per-tenant isolation.
- **Connection Manager** (`server/lib/db-manager.ts`): `TenantDbManager` class manages per-tenant MongoDB connections with:
  - In-memory connection cache (`Map<string, CachedConnection>`)
  - `getTenantConnection(tenantId, dbUri)` — returns cached or creates new `mongoose.createConnection()` with pool (2-10)
  - Automatic idle cleanup: connections unused for 30 minutes are closed (checked every 5 minutes)
  - `hasConnection(tenantId)`, `getActiveConnectionCount()`, `closeAll()` utility methods
  - Singleton export: `tenantDbManager`
- **Model Factories**: `getMessageModel(conn)`, `getConversationModel(conn)`, `getCustomerModel(conn)`, and `getWhatsAppTemplateModel(conn)` in their respective model files use the safe `conn.models[Name] || conn.model(Name, Schema)` pattern to avoid OverwriteModelError. Static exports remain for backward compatibility with unmigrated services.
- **Auth Middleware Injection** (`server/middleware/auth.middleware.ts`): `requireAuth` extends `Express.Request` with `tenantDbConnection?: mongoose.Connection`. After validating user, resolves URI via `resolveTenantDbUri()` with 5-min TTL in-memory cache + pending dedup (thundering herd protection). ## Real-Time Architecture
- **MongoDB Change Streams** (`server/services/change-stream.service.ts`): Watches `messages` and `conversations` collections for inserts/updates. Emits socket events regardless of which server instance (dev vs production) processed the webhook. This solves the multi-instance problem where webhooks hit production but dev server needs real-time updates.
- **Double-emit prevention**: `markLocalEmit(msgId)` in change-stream service prevents re-emitting events already emitted locally by the webhook processor (dedup via 10s TTL Set).
- **Socket rooms**: `new-message` → `conversation:${id}` room; `conversation-updated` → `tenant:${tenantId}` room.
- **queryClient**: `staleTime: Infinity` — all updates MUST use `invalidateQueries` or `setQueryData`; tab-counts need explicit invalidation separately.
- **Superadmin tenant access**: Superadmin has a dummy `tenantId` that doesn't match real tenant IDs. Endpoints must skip tenantId filtering for superadmin role (e.g., mark-read, conversation queries).
